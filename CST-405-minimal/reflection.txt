Compiler Reflection

The most challenging feature to implement was control-flow integration, especially making if/else, nested if blocks, and switch/case/default/break all behave correctly across the full pipeline. At first, each piece looked manageable on its own: scanner keywords, parser rules, AST nodes, TAC labels, and MIPS branches. The hard part was that small mistakes in one phase showed up as confusing behavior in another. Two examples stood out. First, resolving the dangling-else ambiguity required careful grammar precedence so else bound to the nearest if, not an outer one. Second, switch and break handling forced us to model context explicitly, because break had to target the innermost valid construct while still supporting nested structures. The project really proved that control flow is less about syntax and more about preserving exact semantics through every intermediate representation.

Our optimizer mainly works on TAC and applies practical, correctness-focused passes: constant folding, copy propagation, and branch cleanup (including dead branch and dead loop elimination when conditions are compile-time constants). It also tracks loop boundaries so that aggressive substitutions are limited to safe regions. That loop-awareness mattered because values that look stable outside loops can become stale inside loops due to mutation. By folding arithmetic and relational expressions early, we replace multi-instruction computations with direct assignments to literals. By propagating copies, we reduce temporary churn and simplify downstream instruction selection. By removing unreachable branch paths when conditions are known, we generate cleaner control flow with fewer jumps. These reductions improve output in two ways: generated MIPS is smaller and simpler, and the compiler itself runs faster because later phases process less TAC. Our benchmarking summary showed that these and related improvements produced about a 38% reduction in total compile time and a similar drop in memory delta, which confirms the optimization work was not only theoretically correct but practically valuable.

If I were starting over, I would formalize intermediate invariants and test harnesses earlier, especially around control-flow correctness and optimizer safety. We added good tests, but many came after painful debugging sessions. I would begin with a small suite that validates AST shape, TAC label structure, and branch behavior for nested constructs before expanding features. I would also define explicit pass contracts (for example, when propagation is allowed, when loop depth must invalidate assumptions) so later optimizations are easier to reason about. That upfront discipline would have reduced rework and made integration smoother.
