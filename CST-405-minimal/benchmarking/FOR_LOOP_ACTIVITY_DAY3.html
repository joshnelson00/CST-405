E.1 Design loop invariant code motion (LICM)

Loop invariant code motion moves computations whose operands do not change across iterations outside the loop — into the init section — so they execute once instead of N times. Consider this loop:

for (i = 0; i < n; i = i + 1) {
    result = result + base * scale;  // base and scale are loop-invariant
}

The subexpression base * scale is computed on every iteration even though neither operand changes. If LICM were implemented:

  1. How would the TAC for the loop change? Show the before and after TAC for the body and the init.
  2. What data structure would the optimizer need to maintain to determine that a TAC operand is loop-invariant? What information about each variable is required?
  3. At what point in the TAC linked list would the hoisted instruction be inserted? Is the current TAC data structure in tac.h/tac.c capable of insertion at an arbitrary position?
  4. How would LICM interact with the three labels in the for-loop TAC? Specifically, which of the three labels (start, update, end) marks the boundary above which code can be safely hoisted?

Using your metrics from Task 3.1, estimate how many MIPS instructions LICM would save for a loop body containing two loop-invariant multiplications.

E.2 Strength reduction analysis

Strength reduction replaces expensive operations (multiplication, division) with cheaper ones (addition, shift) when the relationship between loop iteration and the computed value is linear. For a loop computing result = i * 4, a strength-reduced version replaces the multiply with an accumulator: initialize t = 0 before the loop, then t = t + 4 in the update step.

Using your MIPS mapping from Task 2.1, answer:

  1. How many MIPS instructions does a TAC_BINOP multiply generate compared to a TAC_BINOP add? What is the instruction-count saving per iteration for a loop of N=10 iterations?
  2. Strength reduction introduces a new induction variable (the accumulator). In the TAC, this requires an additional variable declaration and an additional TAC_ASSIGN in the init. Does the per-iteration saving outweigh the one-time setup cost? At what minimum loop count does strength reduction become beneficial?
  3. The current compiler does not implement strength reduction. Based on your data from Program B (nested loops), estimate the potential MIPS_TOTAL reduction if the inner loop body's multiplications were strength-reduced. Show your calculation.
