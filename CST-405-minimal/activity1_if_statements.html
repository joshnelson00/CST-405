Lexer — Recognising if and else

scanner.l · estimated 10 min

⏱ 10 min
Phase 1 · Lexer

The lexer reads raw characters and groups them into tokens. Currently it knows about while and for but not if or else. When the lexer does not recognise a keyword it falls through to the identifier rule and returns an ID token, which confuses the parser.

✏ Task 1.1 — Add IF and ELSE tokens to scanner.l

  1. Open scanner.l.
  2. Find the section where keywords like while and for are defined.
  3. Add two new rules — place them before the identifier catch-all ([a-zA-Z_][a-zA-Z0-9_]*).
/* ── scanner.l changes ── */
/* These already exist — shown for context */
"while"   { return WHILE; }
"for"     { return FOR;   }

/* ADD THESE TWO LINES ↓ */
"if"      { return IF;    }   /* if keyword */
"else"    { return ELSE;  }   /* else keyword */

✏ Task 1.2 — Declare the tokens in parser.y

The parser must know about the new tokens. Find the %token line and add IF ELSE:

/* parser.y — %token declarations */
/* Existing line — shown for context */
%token INT PRINT RETURN WHILE FOR
%token IF ELSE    /* new tokens for if-statements */

💡 Why Order Matters in Flex

Flex applies rules in declaration order when multiple rules can match the same string. If the identifier rule appeared before "if", then the string if would match the identifier pattern first and be returned as an ID token. By listing keywords before the identifier rule, keywords always win.

✏ Task 1.3 — Build and test the lexer

make clean && make 2>&1 | grep -E "(error|warning|conflict)"

Expected: no errors, no conflicts yet.
Then test the lexer in isolation:

echo "if else" | ./minicompiler /dev/stdin /tmp/out.s 2>&1 | head -5

You should now get a Syntax Error (parser doesn't understand if yet) rather than an identifier token. Good — we broke it in a useful way. The next part fixes it.

Parser — The if_stmt Grammar Rule

parser.y · estimated 20 min

⏱ 20 min
Phase 2 · Parser

2.1 Understanding the ambiguity

Before writing any code, consider this program:

if (a)
    if (b)
        x = 1;
    else
        x = 2;

✏ Task 2.1 — Discussion (5 min)

With your partner, answer:

  1. Draw the two possible parse trees for this program. Which one is "correct" by the C standard?
  2. Why does the naive grammar rule create an ambiguity?
    Hint: After parsing IF(a) stmt, the parser sees ELSE. Should it reduce the outer if or keep reading?
  3. What is Bison's default behaviour when there is a shift/reduce conflict? Does this happen to produce the correct parse?

2.2 Adding the grammar rule

✏ Task 2.2 — Add if_stmt to parser.y (step by step)

Step A — Add precedence declarations before the operator precedences:

/* Add BEFORE the %left/%right operator declarations */
%nonassoc LOWER_THAN_ELSE  /* pseudo-token — never in scanner */
%nonassoc ELSE             /* higher precedence → bison shifts ELSE */

Step B — Add if_stmt to the non-terminal type list:

/* Find the %type declarations and add if_stmt */
%type <node> stmt while_stmt for_stmt  /* existing */
%type <node> if_stmt                   /* new */

Step C — Add if_stmt to the stmt rule:

stmt:
    decl
  | assign
  | while_stmt
  | for_stmt
  | if_stmt          /* new */
  | block
  ;

Step D — Write the if_stmt production rule:

if_stmt:
    IF '(' expr ')' stmt %prec LOWER_THAN_ELSE
        {
            /* if without else */
            $$ = createIf($3, $5, NULL);
        }
  | IF '(' expr ')' stmt ELSE stmt
        {
            /* if with else */
            $$ = createIf($3, $5, $7);
        }
  ;

💡 Reading the %prec Annotation

%prec LOWER_THAN_ELSE tells Bison: "When deciding whether to reduce this rule or shift the next token, compare the precedence of the shift (ELSE) against LOWER_THAN_ELSE." Since ELSE has higher precedence, the shift wins — the else is consumed and attached to the nearest if.

✏ Task 2.3 — Build and check for shift/reduce conflicts

make clean && make 2>&1

Expected: zero shift/reduce or reduce/reduce conflicts. If you see a conflict warning, double-check that:

  • %nonassoc LOWER_THAN_ELSE appears before %nonassoc ELSE in the precedence block.
  • %prec LOWER_THAN_ELSE is on the first production (no else), not the second.

AST — NODE_IF and createIf()

Phase 3 · AST

The parser calls createIf() but we haven't written it yet — the build will fail with "undefined reference to createIf". Now we add the AST support.

✏ Task 3.1 — Add NODE_IF to ast.h (enum)

/* In ast.h — NodeType enum, after NODE_FOR */
NODE_FOR,           /* existing */
NODE_IF,            /* new: if (condition) then [else alt] */

✏ Task 3.2 — Add the if_stmt data field to the union in ast.h

Find the union inside struct ASTNode and add:

/* In ast.h — ASTNode union, after the while_stmt struct */
        /* If statement (NODE_IF) */
        struct {
            struct ASTNode* condition;  /* boolean expression */
            struct ASTNode* then_stmt;  /* taken when condition is true */
            struct ASTNode* else_stmt;  /* taken when false (NULL = no else) */
        } if_stmt;

✏ Task 3.3 — Declare createIf() in ast.h

/* In ast.h — function declarations section */
ASTNode* createIf(ASTNode* condition,
                   ASTNode* then_stmt,
                   ASTNode* else_stmt);

✏ Task 3.4 — Implement createIf() in ast.c

Add this function after the existing createFor() or createWhile():

/* Create an if-statement AST node.
 * else_stmt is NULL when there is no else clause. */
ASTNode* createIf(ASTNode* condition,
                   ASTNode* then_stmt,
                   ASTNode* else_stmt) {
    ASTNode* node = malloc(sizeof(ASTNode));
    node->type   = NODE_IF;
    node->lineno = yylineno;
    node->data.if_stmt.condition = condition;
    node->data.if_stmt.then_stmt = then_stmt;
    node->data.if_stmt.else_stmt = else_stmt;
    return node;
}

✏ Task 3.5 — Add NODE_IF to printAST() in ast.c

Find the switch(node->type) in printAST() and add a case for NODE_IF:

        case NODE_IF:
            printf("IF\n");
            /* Print condition */
            for (int i = 0; i < level + 1; i++) printf("  ");
            printf("CONDITION:\n");
            printAST(node->data.if_stmt.condition, level + 2);
            /* Print then-branch */
            for (int i = 0; i < level + 1; i++) printf("  ");
            printf("THEN:\n");
            printAST(node->data.if_stmt.then_stmt, level + 2);
            /* Print else-branch only if it exists */
            if (node->data.if_stmt.else_stmt) {
                for (int i = 0; i < level + 1; i++) printf("  ");
                printf("ELSE:\n");
                printAST(node->data.if_stmt.else_stmt, level + 2);
            }
            break;

Semantic Analysis — Checking If-Statements

semantic.c · estimated 20 min

⏱ 20 min
Phase 4 · Semantic

The semantic analyzer walks the AST and verifies program correctness. Right now it has no NODE_IF case, so it silently ignores if-statements — any variable reference inside an if-block will not be checked.

✏ Task 4.1 — Add the NODE_IF case to checkStmt()

In semantic.c, find the switch(node->type) inside checkStmt() and add:

        case NODE_IF:
            printf("  ✓ Checking if statement at line %d",
                   node->lineno);

            /* Report whether an else branch is present */
            if (node->data.if_stmt.else_stmt) {
                printf(" (with else)\n");
            } else {
                printf(" (no else)\n");
            }

            /* Step 1: check condition expression for undeclared variables */
            checkExpr(node->data.if_stmt.condition);

            /* Step 2: warn if condition is a compile-time constant */
            if (node->data.if_stmt.condition &&
                node->data.if_stmt.condition->type == NODE_NUM) {
                int v = node->data.if_stmt.condition->data.num;
                fprintf(stderr,
                    "Warning (line %d): if condition is always %s\n",
                    node->lineno,
                    v ? "true (else is dead code)"
                      : "false (then-body is dead code)");
                semInfo.warningCount++;
            }

            /* Step 3: check the then-branch */
            checkStmt(node->data.if_stmt.then_stmt);

            /* Step 4: check the else-branch only if present */
            if (node->data.if_stmt.else_stmt) {
                checkStmt(node->data.if_stmt.else_stmt);
            }
            break;

🔧 Common Mistakes

  • Forgetting %prec LOWER_THAN_ELSE on the first production → shift/reduce conflict warning in bison output.
  • Putting ELSE before LOWER_THAN_ELSE in the %nonassoc list → else attaches to the outer if (wrong!).
  • Not adding NODE_IF to the printAST switch → silent crash or garbage output.
  • Forgetting the NULL check on else_stmt before calling checkStmt → null pointer dereference.
